# 보안 체크리스트 (유료 서비스 운영 기준)

이 문서는 로그인/회원가입 및 예약 시스템을 유료 서비스로 운영할 때 필요한 보안·운영 기준을 정리합니다.
항목을 완료했으면 체크박스를 표시하고, 변경이 있을 경우 변경 사유와 날짜를 남깁니다.

## 인증/권한

- [ ] 관리자 권한은 `NEXT_PUBLIC_ADMIN_EMAIL` 의존을 제거하고, DB 기반 role/claim으로 관리한다.
- [ ] 이메일 인증을 필수로 한다.
- [ ] 비밀번호 정책(최소 길이/복잡도)을 적용한다.
- [ ] 비밀번호 재설정 시도 제한 및 쿨다운 정책이 있다.
- [ ] 로그인 실패 횟수 제한 또는 CAPTCHA를 적용한다.
- [ ] 세션 만료 시간/재로그인 정책을 정의한다.
- [ ] 다중 기기 로그인 관리(기기 로그아웃) 정책이 있다.

## 데이터 접근 제어 (RLS)

- [ ] `profiles` 는 본인 조회/수정만 가능하도록 RLS가 검증된다.
- [ ] `bookings` 는 본인 예약만 조회/수정 가능하도록 RLS가 검증된다.
- [ ] `time_slots` 는 공개 범위(조회 가능/관리자 전용)를 명확히 분리했다.
- [ ] 관리자 전용 정책이 별도로 존재하며, 일반 사용자와 분리 검증된다.
- [ ] 서비스 계정/서버 API 키 사용 범위를 최소화한다.

## 예약/횟수 정합성

- [ ] 예약 생성/취소/자동승격(대기) 로직이 트랜잭션으로 묶인다.
- [ ] 중복 예약 방지 로직이 경합 상황에서 안전하다.
- [ ] 횟수 차감/복구가 원자적으로 처리된다.
- [ ] 예약/취소 정책 변경 시 반영 범위와 시점이 명확하다.

## 감사 로그/추적

- [ ] 관리자 변경(예약 취소/수정/횟수 변경) 기록이 남는다.
- [ ] 자동 처리(노쇼/자동승격) 기록이 남는다.
- [ ] 정책 변경 이력이 남는다.

## 개인정보/법적 준수

- [ ] 개인정보 처리방침을 제공한다.
- [ ] 수집 목적, 보관 기간, 파기 정책을 명시한다.
- [ ] 회원 탈퇴 시 데이터 처리(익명화/삭제) 정책이 있다.

## 운영/알림

- [ ] 예약/취소 정책 변경 시 사용자 고지 방식이 있다.
- [ ] 이상 징후(로그인 실패 급증, 대량 취소 등) 모니터링 체계가 있다.
